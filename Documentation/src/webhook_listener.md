# Enterprise Architecture & Functional Specification: `src/webhook_listener.py`

## 1. Executive Overview

The `src/webhook_listener.py` module acts as the **High-Velocity Ingestion Gateway** for the Intelligence Fusion Center (IFC). Operating as an independent, asynchronous FastAPI microservice within its own dedicated Docker container (`webhook`), it is responsible for receiving, parsing, and normalizing raw telemetry payloads from external Network Management Systems (NMS) and ITSM platforms like SolarWinds, Datadog, or PRTG.

By strictly decoupling ingestion from the core Streamlit UI and background worker daemons, this architecture ensures that sudden "alert storms" (e.g., hundreds of network devices alarming simultaneously during a regional power outage) do not cause UI latency, backend scheduler stalling, or database locking.

---

## 2. Core Architecture: The Asynchronous Gateway

### 2.1 FastAPI & ASGI Execution
* **Framework:** Built on the modern FastAPI framework and served via Uvicorn (exposed on port `8100`), utilizing Python's native `async/await` syntax to handle thousands of concurrent HTTP POST requests with a highly optimized memory footprint.
* **Transient Database Sessions:** Uses SQLAlchemy `Depends(get_db)` dependency injection to open and close isolated, short-lived database transactions for each incoming webhook. This prevents connection pool exhaustion during massive alert floods.

### 2.2 Endpoint Routing
* **`/webhook/solarwinds`:** The primary active POST endpoint. It is designed to accept deeply nested, proprietary JSON payloads natively generated by SolarWinds alert actions.
* **Health Check (`/health`):** A lightweight GET endpoint utilized by Docker Compose to verify container readiness and ensure the ingestion API is actively listening before routing traffic.

---

## 3. Telemetry Normalization & Payload Extraction

External monitoring tools often send non-standardized JSON depending on what triggered the alert (e.g., a node going down vs. a high-CPU warning). The listener forces these varied structures into a unified enterprise schema.

### Data Safing & Failsafes
* **KeyError Mitigation:** Extensively utilizes `.get()` and nested dictionary traversal to safely extract critical NOC metrics, ensuring the API returns a `200 OK` rather than crashing if an expected custom property is missing.
* **Extracted Variables:** * `node_name`, `ip_address`, `status`, and `event_category`.
    * **Performance Metrics:** Safely extracts `CPULoad` and `PercentLoss`, stripping out arbitrary string characters (like `%`) and casting them to strict integers/floats for the AIOps engine to run threshold math against later.

---

## 4. Heuristic Asset Mapping & Fingerprinting

Before an alert is committed to the database, the gateway attempts to map the digital failure to a physical `MonitoredLocation` to support the geospatial RCA engine.

* **Explicit Tagging (Primary):** Evaluates explicit metadata from the NMS (e.g., extracting the `Site` or `District` fields directly from `Custom_Properties_Universal`).
* **String Fingerprinting (Secondary):** If explicit tags are missing or malformed, it utilizes string heuristics and substring matching against the device's raw hostname to identify aliases corresponding to known physical facility names.
* **Fallback State:** If a node cannot be confidently mapped to a physical location, its site is flagged as `"Unknown"`. This ensures the raw alert is still captured for IT logging without falsely skewing the PyDeck geospatial map.

---

## 5. Persistence & AIOps Handoff

The listener is strictly an ingestion tool; it intentionally performs zero analytical operations to preserve its high-speed I/O capability.

* **Database Commits (`SolarWindsAlert`):** The standardized data is instantiated as a `SolarWindsAlert` ORM object and committed to the database. The `raw_payload` (the unparsed original JSON) is saved into a distinct column to support future ML retraining, forensic auditing, or custom playbook generation.
* **Correlation Flags:** The alert is explicitly saved with `is_correlated = False` and `is_dispatched = False`.
* **The Decoupled Handoff:** Once the row is committed, the listener's job is done. The `EnterpriseAIOpsEngine`—running asynchronously in the separate `worker` container—polls the database every 5 seconds to scoop up these un-correlated alerts, calculate their topological blast radii, and cluster them into physical incidents.
